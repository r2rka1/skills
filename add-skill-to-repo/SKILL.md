# Add Skill to Repository

Creates a new skill at the **repository root** of the skills repo (`/home/r2rka/projects/skills/<skill-name>/`), making it a versioned, shareable skill.

This differs from `/create-skill` which creates skills under `.claude/skills/` for local use. Skills created by this command live in the repo root and are tracked by git.

## Usage

```
/add-skill-to-repo my-new-skill
```

## Workflow

### 1. Validate Input

- [ ] Ensure a skill name was provided as `$ARGUMENTS[0]`. If not, ask the user.
- [ ] Validate the name: lowercase letters, numbers, and hyphens only (max 64 chars).
- [ ] No reserved words ("anthropic", "claude").
- [ ] Confirm `/home/r2rka/projects/skills/$ARGUMENTS[0]` does not already exist.

### 2. Ask the User

Ask the user:
- What should the skill do?
- Should it be user-invocable (slash command), model-invocable (Claude triggers automatically), or both?

Default: user-invocable only.

### 3. Create Directory Structure

```bash
mkdir -p /home/r2rka/projects/skills/$ARGUMENTS[0]/scripts
mkdir -p /home/r2rka/projects/skills/$ARGUMENTS[0]/resources
```

### 4. Generate SKILL.md

Create `/home/r2rka/projects/skills/$ARGUMENTS[0]/SKILL.md` with:

**Frontmatter:**

```yaml
---
name: <skill-name>
description: <what it does and when to use it — third person, specific, includes key terms>
---
```

**Required body sections** (always include these):

```markdown
## Directory Structure

- `resources/` — persistent output and data files generated by this skill
- `scripts/` — reusable scripts for this skill's operations

## Script Management

When performing an operation that can be scripted:
1. Check `scripts/` for an existing script that handles this operation
2. If a script exists, execute it instead of doing the work inline
3. If no script exists and the operation is reusable, create one in `scripts/`, make it executable, then execute it
4. Reference any new scripts in this SKILL.md under "Available Scripts"

## Available Scripts

_No scripts yet. Scripts will be added here as they are created._
```

### 5. Content Rules

- Keep SKILL.md body under 500 lines
- Write description in third person
- Be concise — only include context Claude doesn't already have
- Use progressive disclosure: link to supporting files for detailed content
- Provide concrete examples, not abstract descriptions
- Use consistent terminology
- For multi-step workflows, include a checklist pattern
- For output formatting, include a template pattern

### 6. Frontmatter Options

Available optional fields:

| Field | Purpose |
|-------|---------|
| `disable-model-invocation: true` | Only user can invoke |
| `user-invocable: false` | Only Claude can invoke |
| `allowed-tools: Read, Grep, Glob` | Restrict tool access |
| `context: fork` | Run in isolated subagent |
| `agent: Explore` | Subagent type when using `context: fork` |
| `argument-hint: "[args]"` | Autocomplete hint |
| `model: sonnet` | Override model |

### 7. Verify and Report

- [ ] Show the user the full SKILL.md content
- [ ] Confirm the directory structure exists (`scripts/`, `resources/`)
- [ ] Remind the user to copy the skill to `.claude/skills/` if they want it active locally:
  ```bash
  cp -r /home/r2rka/projects/skills/<skill-name> /home/r2rka/projects/.claude/skills/
  ```
- [ ] Note the skill can be committed and pushed to the repo for sharing

## Quality Checklist

Before finishing, verify:
- [ ] Description is specific and includes trigger terms
- [ ] Description is written in third person
- [ ] SKILL.md body is under 500 lines
- [ ] `resources/` and `scripts/` directories created
- [ ] Script Management section included in SKILL.md
- [ ] Available Scripts section included in SKILL.md
- [ ] No time-sensitive information
- [ ] Consistent terminology
