# Create README

Analyzes the current project and generates a well-structured `README.md` file based on what it finds.

## Usage

```
/create-readme
```

Run this from the root of a project directory.

## Workflow

### 1. Analyze the Project

Scan the project to gather information:

- [ ] **Project name** — from `package.json`, `Cargo.toml`, `pyproject.toml`, `go.mod`, folder name, or other manifest files
- [ ] **Description** — from manifest file description field if available
- [ ] **Language/framework** — detect from file extensions, config files, and dependencies
- [ ] **Dependencies** — from lock files and manifest files
- [ ] **Scripts/commands** — from `package.json` scripts, `Makefile`, `Taskfile`, etc.
- [ ] **Entry points** — `main.py`, `index.ts`, `src/main.rs`, `cmd/`, etc.
- [ ] **Configuration** — `.env.example`, config files, environment variables
- [ ] **Tests** — test directories, test frameworks, test commands
- [ ] **Docker** — `Dockerfile`, `docker-compose.yml`
- [ ] **CI/CD** — `.github/workflows/`, `.gitlab-ci.yml`, etc.
- [ ] **License** — `LICENSE` file

### 2. Generate README.md

Create a `README.md` with the following sections (include only what's relevant):

```markdown
# Project Name

Brief description of what the project does.

## Features

- Key feature 1
- Key feature 2

## Prerequisites

- Runtime version (e.g., Node.js >= 18)
- Required tools

## Getting Started

### Installation

Step-by-step installation instructions.

### Configuration

Environment variables and config files needed.

### Running

How to start the application.

## Development

### Project Structure

Brief overview of key directories and files.

### Available Scripts/Commands

Table or list of useful commands.

### Testing

How to run tests.

## Deployment

Deployment instructions if Docker/CI config is found.

## License

License type and link.
```

### 3. Adapt to Project Type

Tailor the README based on what's detected:

| Project Type | Extra Sections |
|---|---|
| **Library/package** | API usage examples, installation via package manager |
| **CLI tool** | Command-line usage, flags, examples |
| **Web app** | Tech stack, screenshots placeholder, API endpoints |
| **API service** | Endpoints overview, authentication, request examples |
| **Monorepo** | Package/workspace overview, links to sub-READMEs |

### 4. Quality Rules

- [ ] Use clear, concise language
- [ ] Include actual commands from the project (not generic placeholders)
- [ ] Code blocks should have language identifiers (```bash, ```json, etc.)
- [ ] Keep it practical — focus on getting started quickly
- [ ] Do not fabricate features or capabilities not found in the code
- [ ] If an existing `README.md` exists, ask the user whether to overwrite or update it

### 5. Verify and Report

- [ ] Show the user the generated README content
- [ ] Confirm the file was written to `README.md`
- [ ] Highlight any sections that may need manual additions (screenshots, detailed API docs, etc.)

## Directory Structure

- `resources/` — persistent output and data files generated by this skill
- `scripts/` — reusable scripts for this skill's operations

## Script Management

When performing an operation that can be scripted:
1. Check `scripts/` for an existing script that handles this operation
2. If a script exists, execute it instead of doing the work inline
3. If no script exists and the operation is reusable, create one in `scripts/`, make it executable, then execute it
4. Reference any new scripts in this SKILL.md under "Available Scripts"
