# Refactor Code

Acts as a senior developer performing a systematic refactoring review. Identifies concrete improvement opportunities and applies them with explanations.

## Usage

```
/refactor-code
/refactor-code src/utils/
/refactor-code src/handlers/auth.ts
```

If no path is provided, analyzes recently changed files (`git diff --name-only HEAD~5`).

## Workflow

### 1. Determine Scope

- [ ] If a file or directory path was provided, use that as the scope
- [ ] If no path was provided, identify recently changed files from git history
- [ ] If the scope is a directory, prioritize files by size and complexity (largest/most complex first)
- [ ] Limit analysis to a reasonable scope — no more than 10 files per run

### 2. Analyze Code

Read each file in scope and identify refactoring opportunities in these categories:

| Category | What to Look For |
|----------|-----------------|
| **Duplication** | Repeated code blocks, copy-pasted logic, similar functions that can be unified |
| **Long functions** | Functions over ~40 lines that do multiple things and should be split |
| **Naming** | Unclear variable/function names, inconsistent naming conventions, abbreviations |
| **Complex conditionals** | Deeply nested if/else, long boolean expressions, missing early returns |
| **Dead code** | Unused imports, unreachable branches, commented-out code, unused variables |
| **Abstraction** | Missing or leaky abstractions, god classes/modules, tight coupling |
| **Error handling** | Swallowed errors, inconsistent error patterns, missing edge cases |
| **Type safety** | `any` types, missing null checks, loose type assertions |

### 3. Prioritize Findings

Rank each finding by impact:

1. **High** — Bugs waiting to happen, significant duplication, major readability issues
2. **Medium** — Structural improvements, naming clarity, moderate complexity reduction
3. **Low** — Style preferences, minor simplifications, cosmetic changes

Focus refactoring effort on High and Medium items. Report Low items but do not change them unless the user asks.

### 4. Present Findings

Before making changes, present a summary to the user:

```markdown
## Refactoring Analysis: <scope>

### High Priority
- **<file>:<line>** — <description of issue and proposed fix>

### Medium Priority
- **<file>:<line>** — <description of issue and proposed fix>

### Low Priority (no action taken)
- **<file>:<line>** — <description>

**Proposed changes:** <N> files, <M> refactorings
```

Ask the user for confirmation before applying changes.

### 5. Apply Refactorings

For each approved refactoring:

- [ ] Make the change using the smallest possible edit
- [ ] Preserve existing behavior — refactoring must not change functionality
- [ ] Keep the project's existing code style and conventions
- [ ] Do not add new dependencies
- [ ] Do not change public APIs unless the user explicitly approves

### 6. Verify

After applying changes:

- [ ] Run the project's test suite if one exists (`npm test`, `pytest`, `cargo test`, etc.)
- [ ] If tests fail, revert the change that caused the failure and report it
- [ ] Show a summary of all changes made with before/after comparisons

## Refactoring Principles

- **Preserve behavior** — refactoring changes structure, not functionality
- **Small steps** — apply one refactoring at a time, verify after each
- **Follow existing patterns** — match the project's style, don't impose new conventions
- **Explain the why** — every change should have a clear reason tied to a specific code smell
- **Don't over-abstract** — three similar lines are better than a premature abstraction
- **Respect scope** — only refactor code within the requested scope

## Directory Structure

- `resources/` — persistent output and data files generated by this skill
- `scripts/` — reusable scripts for this skill's operations

## Script Management

When performing an operation that can be scripted:
1. Check `scripts/` for an existing script that handles this operation
2. If a script exists, execute it instead of doing the work inline
3. If no script exists and the operation is reusable, create one in `scripts/`, make it executable, then execute it
4. Reference any new scripts in this SKILL.md under "Available Scripts"
